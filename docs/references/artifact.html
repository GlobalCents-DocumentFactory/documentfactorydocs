Artifact

Artifact is a common name for a schema of objects that manages resources and directories. For example, it can define and manage various external resources (artifacts) embedded into documents during the document processing pipeline or specify input and output directories, facilitating the organization and processing of documents. Below are the properties available for configuration along with their descriptions.

asset​

The property is used to define external resources. The asset property is an object that can have the following properties:

  • storetype - specifies the storage type where the external resource is located.
  • mimetype - specifies the mimetype of the external resource.
  • name - ???.
  • shouldbeprocessed - specifies whether the resource should be processed.
  • artifactfs - configuration details specific to the file system storage type.
  • artifactcs - configuration details specific to the Content Server storage type.
  • artifactnats - configuration details specific to the Nats storage type.

storetype​

Specifies the storage type where the external resource is located. This property enables to correctly access and retrieve the asset based on its storage mechanism. Each storage type corresponds to a different method of accessing and managing the resource, ensuring compatibility with various storage solutions. Can take the following storage type values: none, fs, nats, ftp, s3, sharepoint, contentserver, url.

mimetype​

The mimetype property within the asset object is a string that defines the media type of the external resource. Correctly specifying this property ensures that the Document Factory platform can process and manage the asset properly, regardless of the stage it is used in the document handling workflow.

Common MIME Types:

  • application/pdf: PDF document
  • image/png: PNG image
  • image/jpeg: JPEG image
  • text/plain: Plain text file
  • application/msword: Microsoft Word document
  • application/vnd.openxmlformats-officedocument.wordprocessingml.document: Microsoft Word (OpenXML format)

Example:

pipeline.json
{
  "merge": true,
  "mergesettings": {
    "saveformat": "pdf",
    "mimetype": "application/pdf",
    "intermediatepage":{
      "asset": {
        "storetype": "fs",
        "artifactfs": {
          "fullname": "C:\assets\Separator.pdf"
        }
      }
    }
  }
}

shouldbeprocessed​

The shouldbeprocessed property within the asset object is a boolean that specifies whether the resource should be processed during the document handling workflow. Setting this property to true indicates that the resource should be included in the processing operations, while setting it to false excludes the resource from being processed.

artifactfs​

The artifactfs is used to provide additional configuration details specific to the file system storage type when storetype is set to fs. This property ensures that all necessary information regarding file locations and access methods are specified, allowing the DocumentFactory to correctly locate and handle assets stored on a file system. The artifactfs configuration includes the following settings:

fullname​

The fullname property within the artifactfs configuration specifies the complete file path to the asset when the file system storage type (fs) is used. It provides a direct way to define the exact location of the asset on the file system.

Example:

pipeline.json
{
  "merge": true,
  "mergesettings": {
    "saveformat": "pdf",
    "intermediatepage":{
      "asset": {
        "storetype": "fs",
        "artifactfs": {
          "fullname": "C:\assets\Separator.pdf"
        }
      }
    }
  }
}

fsdocumentsavestrategy​

The fsdocumentsavestrategy defines the strategy to be used for saving documents to the file system when the fs (file system) storage type is chosen. This property provides flexibility in determining how the documents are stored, managed, and accessed on the file system, ensuring that the saving mechanism aligns with the desired organizational and operational practices. The 'fsdocumentsavestrategy' configuration is defined by the following properties:

  • keepfolderstructure - specifies whether the original folder structure of the source documents should be maintained.
  • namingpattern - defines the pattern to be used for naming saved documents.
  • overwrite - specifies whether existing files should be overwritten when saving a new document with the same name.
  • replaceoriginalextension - specifies whether the original file extension should be replaced with a standard extension when saving the document.
  • serverside - specifies whether the document saving operations should be performed on the server side.
keepfolderstructure​

Property specifies whether the original folder structure of the source documents should be maintained when saving them to the file system. When set to true, the original directory structure from the source location is preserved in the destination directory.

namingpattern​

Property defines the pattern to be used for naming saved documents. You can find more detailed information about the use of namingpattern with examples in the section Naming Pattern.

overwrite​

Property specifies whether existing files should be overwritten when saving a new document with the same name to the file system. When set to true, any existing file with the same name at the destination will be overwritten by the new file. When set to false, it prevents overwriting and can be combined with namingpattern to generate unique filenames.

replaceoriginalextension​

Property specifies whether the original file extension should be replaced with a standard extension when saving the document. When set to true, the original file extension is replaced with a default or standard extension (such as .docx, .pdf, etc.) as defined by the saving strategy or system configuration. When set to false, the original file extension is retained, ensuring that the saved document keeps its initial format.

serverside​

Property specifies whether the document saving operations should be performed on the server side. This is relevant for scenarios where the document processing and saving workflows are managed by a server. When set to true, all save operations are carried out on the server, which can be beneficial for performance and security reasons as it minimizes client-side operations.

ordering​

The ordering property specifies the order in which the artifact should be processed relative to other artifacts. This integer value is used to sequence the processing of multiple artifacts, ensuring they are handled in the defined order during document processing operations.

Value Guidelines:

  • Use sequential integers starting from 1 to define the processing order.
  • Ensure that no two artifacts have the same ordering value to avoid conflicts or undefined behavior.
  • Higher ordering values represent later positions in the sequence.

level​

The level property is an integer that defines the hierarchical level or priority of the artifact. This property allows users to organize and manage artifacts based on their relative importance or their place in a hierarchical structure within the document processing workflow. It adds an additional dimension of organization beyond the ordering property, which sequences artifacts.

Value Guidelines:

  • Lower Values (e.g., 1): Indicate higher importance or higher-level priority.
  • Higher Values: Indicate lower importance or lower-level priority.
  • Ensure that the level values are assigned logically to reflect the actual hierarchical structure or priority of the artifacts.

children​

The children property within the Artifact type defines a hierarchical structure by containing an array of child artifacts. This allows for the nesting of multiple Artifact within a parent Artifact, enabling complex document structures and processing workflows that mirror hierarchical or nested relationships. This setup is useful for organizing documents with nested sections, chapters, or components that need to be processed in a hierarchical or nested order.

note
  • Each item in the children array should be a valid Artifact object.
  • The children array can be empty ([]) if there are no child artifacts.
  • The hierarchical relationship should be meaningful and reflect the actual structure required for processing.
